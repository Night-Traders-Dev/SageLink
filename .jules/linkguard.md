# LinkGuard Journal

## Security patterns
- **Hallucinated Vulnerabilities:** Previous audits contained hallucinated vulnerabilities (e.g. unbounded frame length allocation, replay window unauthenticated state mutation, rekey deadlocks) due to assumptions made without tracing actual code implementation. It is verified via tracing `src/mux/stream.sage` that rekey deadlocks are safely prevented by `mux["rekeying"]` logic.
- **Evidence-Based Verification:** It is critical to strictly trace execution paths, verify assumptions with codebase evidence (such as grep and precise reading), and never invent theoretical issues. Always prioritize the repository as the sole source of truth.

## Recurring vulnerabilities
- **Insecure Default Permissions (TOCTOU):** File creation with default permissions followed by a `chmod` creates a race condition for sensitive files (e.g., `identity.key` generated by `keygen`). Validated to still be present in `src/cli/sagelink.sage`.
- **Integrity Check Bypass via Loop Exit:** In `src/app/file.sage`, a malicious chunk that pushes `bytes_written` strictly greater than `file_size` causes the receive loop to exit and entirely bypasses the `bytes_written == file_size` integrity block without deleting the invalid file.
- **Incomplete Malicious File Cleanup:** Failing integrity checks overwrites files to 0-bytes instead of using an unlink/delete syscall, which may lead to clutter or unexpected behaviors if the empty file remains.

## Performance bottlenecks
- **Memory Exhaustion Pattern (OOM):** The `io.readbytes` method was found loading whole files into memory at once for hash verification in `src/app/file.sage`, leading to Out-of-Memory exhaustion when handling large files. Whole-file buffering scales poorly on constrained devices.
- **List Copying Overhead:** Extensive use of element-by-element list copying (e.g. `push()` in loops) instead of native memory operations (due to SageLang constraints) introduces O(N) CPU overhead in the transport layer and multiplexing paths. This is a recurring performance bottleneck.
- **Multiplexer ID Resolution:** Stream ID assignment uses a linear probe with up to 65536 retries in `mux_open_stream`. While it avoids collisions, this is an O(N) algorithm that can degrade if many streams are concurrently open.
- **Excessive I/O Overhead:** Using `io.appendbytes` for every single file chunk heavily degrades I/O performance on large files due to thousands of repeated open/append/close cycles.
- **Polling Loop CPU Overhead:** Code using `thread.sleep(0.005)` in tight while loops (like waiting for stream messages) wastes CPU cycles on embedded devices; a proper blocking condition variable or channel mechanism would be more efficient.

## Architectural weaknesses
- **Hardcoded Command Exit Code:** In `src/app/cmd.sage`, the command service always returns an exit code of `0` (`let resp = [0]`) back to the client, effectively masking actual command failures.
- **Cross-Platform Limitations:** Hardcoded FFI dependency paths (e.g. `libc.so.6` or `libc.so`) cause total breakage on non-Linux systems, such as macOS, which rely on `.dylib` file extensions.
- **Hardcoded IOCTL Values:** The SHELL service uses hardcoded, platform-specific integer IOCTL values (e.g. `TIOCSCTTY = 21518`, `TIOCSWINSZ = 21524` in `src/app/shell.sage`) rather than dynamically retrieving them, preventing cross-platform terminal compatibility.
- **IPv6 Address Parsing Failure:** In `src/cli/sagelink.sage`, the `parse_addr` function unconditionally splits the address on the first colon (`:`). This breaks IPv6 addresses which contain multiple colons.
- **Unauthenticated Queue Hallucination:** Previous audits assumed streams had unauthenticated queues. Streams are multiplexed only *after* the Noise_IK handshake establishes an authenticated session.
- **Path Traversal Hallucination:** The FILE service's sanitization logic (`if c == '/' or c == '\\': filename = ''`) effectively strips all parent directory components by extracting the basename, making it safe from traversal attacks. Previous assumptions about `..` vulnerability were incorrect.

## Reliability risks
- **Unbounded Thread Spawn:** `tcp.accept` triggers `thread.spawn(handle_client)` without limits in `src/cli/sagelink.sage`, allowing unauthenticated attackers to cause thread exhaustion. Additionally, the stream dispatcher spawns threads (`run_cmd`, `run_file`, `run_shell`) without limits for authenticated peers.
- **Unbounded Stream Queue (Memory Exhaustion):** In `src/mux/stream.sage`, incoming messages are blindly pushed to `stream["queue"]` without any backpressure mechanism. If a stream consumer processes messages slower than they arrive, this queue will grow unboundedly, leading to OOM.

## Build system pitfalls
- Always ensure SageLang submodules are checked out correctly to verify `python3 sagemake test` and `python3 sagemake check` execution. Setting up `SAGE_PATH` allows testing locally without system installation. Running tests blindly may cause timeouts. Tailoring tests to exactly what was modified ensures faster validation.
